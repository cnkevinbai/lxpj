## 1. 设计理念

### 1.1 唐朝三省六部制的精髓

```
┌─────────────────────────────────────────────────────────────┐
│                        核心设计原则                          │
├─────────────────────────────────────────────────────────────┤
│  1. 分权制衡 — 决策、审议、执行三权分立，互相制约            │
│  2. 专业分工 — 六部各司其职，术业有专攻                      │
│  3. 流程闭环 — 起草→审核→执行→反馈，形成完整链路            │
│  4. 封驳机制 — 门下省有权驳回不合理诏令，保证质量            │
│  5. 政事堂议 — 三省长官共议国政，集体决策                    │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 现代协作系统的痛点

| 痛点 | 唐朝解决方案 | 现代映射 |
|-----|-------------|---------|
| 权责不清 | 三省各司其职 | Agent角色明确分工 |
| 质量失控 | 门下封驳 | 审核Agent把关 |
| 执行混乱 | 六部专业分工 | 专业Agent群执行 |
| 决策草率 | 政事堂共议 | 多Agent协商机制 |
| 流程断裂 | 诏令流转程序 | 消息队列+状态机 |

### 1.3 设计目标

```yaml
核心目标:
  分权: 决策、审核、执行三权分离，避免单一Agent权力过大
  制衡: 审核Agent有权驳回决策，执行Agent可反馈问题
  专业: 六部Agent各司其职，专业人做专业事
  闭环: 任务从需求到交付全程可追溯
  灵活: 支持单省运作、三省联动、多场景适配
```

## 4. 三省详细设计

### 4.1 中书省 (决策中心)

**职责**: 接收用户需求，分析、设计、规划，输出可执行方案

#### 4.1.1 Agent组成

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **需求分析Agent** | 理解用户意图，提取关键需求 | 用户原始需求 | 需求文档、用户故事 |
| **架构设计Agent** | 技术选型、系统架构设计 | 需求文档 | 架构图、技术方案 |
| **方案规划Agent** | 任务拆解、排期规划 | 架构方案 | 任务清单、时间线 |
| **方案编写Agent** | 整合输出完整方案书 | 以上所有 | 方案文档 |

#### 4.1.2 工作流程

```
用户需求 ──▶ 需求分析Agent ──▶ 需求文档
                                    │
                                    ▼
                            架构设计Agent ──▶ 架构方案
                                    │
                                    ▼
                            方案规划Agent ──▶ 任务清单
                                    │
                                    ▼
                            方案编写Agent ──▶ 完整方案
                                    │
                                    ▼
                              提交门下省审核
```

#### 4.1.3 决策协议

```typescript
interface ZhongshuProposal {
  id: string;                    // 方案ID
  title: string;                 // 方案标题
```

### 4.2 门下省 (审核中心)

**职责**: 审核中书省方案，有权封驳退回，确保质量

| **代码审查Agent** | 代码质量、规范 | 代码风格、安全、性能 | 要求修改 |
| **风险评估Agent** | 技术风险、业务风险 | 风险识别、影响分析 | 风险预警 |
| **质量验收Agent** | 交付标准、测试覆盖 | 验收标准、测试报告 | 验收通过/拒绝 |

## 5. 六部详细设计

### 5.1 吏部 Agent (人力协调)

**职责**: 任务分配、负载均衡、能力匹配

```yaml
能力:
  - 分析任务需求，识别所需技能
  - 评估各Agent当前负载
  - 智能匹配任务与Agent能力
  - 动态调整任务分配

决策规则:
  - 同等条件下，优先分配给负载最低的Agent
  - 关键任务优先分配给能力最强的Agent
  - 支持任务拆分，多人协作

输出:
  - 任务分配方案
  - 负载均衡报告
  - 能力矩阵更新
```

### 5.2 户部 Agent (资源管理)

**职责**: 预算、服务器、第三方服务管理

```yaml
能力:
  - 资源需求预估
  - 成本计算与优化建议
  - 资源申请与释放
  - 第三方服务集成管理

决策规则:
  - 资源申请需在预算范围内
  - 超预算需上报审批
  - 闲置资源及时释放

输出:
  - 资源清单
  - 成本报告
  - 资源使用建议
```

### 5.3 礼部 Agent (文档沟通)

**职责**: 文档编写、对外接口、用户沟通

```yaml
能力:
  - 自动生成技术文档
  - API文档维护
  - 用户通知与公告
  - 变更日志编写

输出:
  - 技术文档
  - API文档
  - 用户公告
  - 变更日志
```

### 5.4 兵部 Agent (核心攻坚)

**职责**: 难点突破、技术预研、关键模块

```yaml
能力:
  - 技术难点攻关
  - 新技术预研
  - 关键模块开发
  - 技术债务清理

决策规则:
  - 优先处理P0级技术问题
  - 预研结果需输出评估报告
  - 关键模块变更需通知刑部测试

输出:
  - 技术方案
  - 核心代码
  - 预研报告
```

## 6. 通信协议

### 6.1 消息格式

```typescript
interface AgentMessage {
  // 消息头
  header: {
    id: string;                  // 消息ID
    from: AgentID;               // 发送者
    to: AgentID | 'BROADCAST';   // 接收者
    type: MessageType;           // 消息类型
    timestamp: Date;             // 时间戳
    correlationId?: string;      // 关联消息ID(用于响应)
  };
  
  // 消息体
  body: {
    action: string;              // 动作
    payload: any;                // 数据
    context?: {                  // 上下文
      proposalId?: string;
      taskId?: string;
      [key: string]: any;
    };
  };
  
  // 元数据
  metadata?: {
    priority: 'LOW' | 'NORMAL' | 'HIGH' | 'URGENT';
    ttl?: number;                // 消息有效期(秒)
    retry?: number;              // 重试次数
  };
}
```

### 6.2 消息类型

```typescript
type MessageType = 
  // 省际通信
  | 'PROPOSAL_SUBMIT'       // 中书提交方案
  | 'PROPOSAL_REVIEW'       // 门下审核结果
  | 'PROPOSAL_APPROVED'     // 方案通过
  | 'PROPOSAL_REJECTED'     // 方案封驳
  | 'TASK_ASSIGN'           // 尚书分配任务
  
  // 六部通信
  | 'RESOURCE_REQUEST'      // 资源请求
  | 'RESOURCE_ALLOCATED'    // 资源分配
  | 'TASK_COMPLETE'         // 任务完成
  | 'ISSUE_RAISE'           // 问题上报
  | 'ISSUE_RESOLVE'         // 问题解决
  
  // 协调通信
  | 'CONSULTATION_REQUEST'  // 咨询请求
  | 'CONSULTATION_RESPONSE' // 咨询回复
  | 'ESCALATION'            // 上报升级
  | 'NOTIFICATION'          // 通知
  
  // 政事堂
  | 'COUNCIL_CONVENE'       // 召集会议
  | 'COUNCIL_VOTE'          // 投票
  | 'COUNCIL_DECISION'      // 会议决议
  ;
```

### 6.3 通信模式

```
┌─────────────────────────────────────────────────────────────┐
│                      通信模式分类                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 请求-响应 (Request-Response)                            │
│     中书 ──[PROPOSAL_SUBMIT]──▶ 门下                        │
│     中书 ◀──[PROPOSAL_REVIEW]── 门下                        │
│                                                             │
│  2. 发布-订阅 (Pub-Sub)                                     │
│     发布者 ──[NOTIFICATION]──▶ 消息队列 ──▶ 订阅者们        │
│                                                             │
│  3. 协作会议 (Meeting)                                       │
│     发起者 ──[COUNCIL_CONVENE]──▶ 多Agent                   │
│     多Agent ──[COUNCIL_VOTE]──▶ 汇总                        │
│     发起者 ──[COUNCIL_DECISION]──▶ 公布结果                 │
│                                                             │
│  4. 工作流 (Workflow)                                        │
│     按预定义流程自动流转消息                                 │

└─────────────────────────────────────────────────────────────┘
```

## 8. 应用场景

### 8.1 多Agent协作系统

```yaml
场景描述:
  多个AI Agent协同完成复杂任务，如自动编程系统、智能客服集群

  中书省: 任务理解、方案生成、Agent角色分配
  门下省: 方案合理性验证、输出质量审核
  尚书省: 各Agent并行执行、结果汇总

示例流程:
  用户: "帮我开发一个博客系统"
  
  中书省:
    - 需求分析Agent: 解析功能需求(文章管理、用户系统、评论)
    - 架构设计Agent: 技术选型(前端React/后端Node)
    - 任务拆解Agent: 分解为前端/后端/数据库任务
  
  门下省:
    - 方案审核Agent: 验证架构合理性
    - 风险评估Agent: 识别技术风险
  
  尚书省:
    - 工部Agent: 编写代码
    - 刑部Agent: 编写测试
    - 户部Agent: 配置数据库
  
  结果: 完整的博客系统代码仓库
```
