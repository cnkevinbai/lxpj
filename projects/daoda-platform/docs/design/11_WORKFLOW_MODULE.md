## 一、模块概述

### 1.1 模块定位

审批工作流模块是企业数字化平台的核心引擎模块，负责业务流程的标准化、自动化执行，支持可视化流程设计、灵活的审批规则配置，实现企业业务流程的高效流转。

## 二、流程模型设计

### 2.1 BPMN 2.0 核心概念

```
┌─────────────────────────────────────────────────────────────────┐
│                    BPMN 2.0 核心元素                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【流程对象】                                                    │
│                                                                 │
│  1️⃣ 事件 (Events)                                               │
│     ┌─────────┐ ┌─────────┐ ┌─────────┐                        │
│     │ 开始事件 │ │ 结束事件 │ │ 中间事件 │                        │
│     │  (圆圈) │ │ (粗圆圈) │ │ (双线圆) │                        │
│     └─────────┘ └─────────┘ └─────────┘                        │
│     • 开始事件: 启动流程                                        │
│     • 结束事件: 结束流程                                        │
│     • 定时器事件: 定时触发                                      │
│     • 消息事件: 消息触发                                        │
│     • 信号事件: 信号触发                                        │
│     • 错误事件: 异常处理                                        │
│                                                                 │
│  2️⃣ 活动 (Activities)                                           │
│     ┌─────────────┐ ┌─────────────┐ ┌─────────────┐            │
│     │   用户任务   │ │   服务任务   │ │   网关      │            │
│     │  (圆角矩形) │ │   (矩形)    │ │  (菱形)    │            │
│     └─────────────┘ └─────────────┘ └─────────────┘            │
│     • 用户任务: 需要人工处理                                    │
│     • 服务任务: 系统自动执行                                    │
│     • 脚本任务: 执行脚本                                        │
│     • 子流程: 嵌套流程                                          │
│                                                                 │
│  3️⃣ 网关 (Gateways)                                             │
│     ┌─────────┐ ┌─────────┐ ┌─────────┐                        │
│     │ 排他网关 │ │ 并行网关 │ │ 包含网关 │                        │
│     │  (X)    │ │  (+)    │ │  (O)    │                        │
│     └─────────┘ └─────────┘ └─────────┘                        │
│     • 排他网关: 条件分支,只走一条路径                           │
│     • 并行网关: 同时执行所有分支                                │
│     • 包含网关: 条件判断,执行符合条件的分支                     │
│     • 事件网关: 等待事件触发                                    │
│                                                                 │
│  4️⃣ 流向 (Flow)                                                 │
│     ┌──────────────────────────────────┐                       │
│     │  ──────────────────────────────▶ │                       │
│     │           顺序流                  │                       │
│     └──────────────────────────────────┘                       │
│     • 顺序流: 连接节点                                          │
│     • 条件顺序流: 带条件的连接                                  │
│     • 默认顺序流: 默认路径                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 审批流程模型

```
┌─────────────────────────────────────────────────────────────────┐
│                    审批流程模型设计                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  【典型审批流程】                                                │
│                                                                 │
│  ┌───────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐    │
│  │ 开始  │────▶│部门经理 │────▶│财务审核 │────▶│总经理   │    │
│  │       │     │ 审批    │     │         │     │ 审批    │    │
│  └───────┘     └─────────┘     └─────────┘     └─────────┘    │
│                                      │                │        │
│                                      │ 金额<1万       │ 通过   │
│                                      │ (跳过)         │        │
│                                      │                ▼        │
│                                      │         ┌─────────┐    │
│                                      └────────▶│ 结束    │    │
│                                                └─────────┘    │
│                                                                 │
│  【审批类型】                                                    │
│                                                                 │
│  1️⃣ 串行审批                                                    │
│     • 按顺序依次审批                                            │
│     • 前一个审批通过后流转到下一个                              │
│     • 适用于层级审批                                            │
│                                                                 │
│  2️⃣ 并行审批                                                    │
│     • 多人同时审批                                              │
│     • 所有人通过才算通过                                        │
│     • 适用于会签                                                │
│                                                                 │
│  3️⃣ 或签审批                                                    │
│     • 多人中的任意一人审批即可                                  │
│     • 一人通过即可流转                                          │
│     • 适用于多人可处理的场景                                    │
│                                                                 │
│  4️⃣ 按比例通过                                                  │
│     • 按通过比例决定                                            │
│     • 如: 50%以上通过即为通过                                   │
│                                                                 │
│  【审批人指定方式】                                              │
│                                                                 │
│  • 指定人员: 固定指定某个用户                                   │
│  • 指定角色: 由具有某角色的用户审批                             │
│  • 指定岗位: 由某岗位的人员审批                                 │
│  • 部门负责人: 由发起人部门负责人审批                           │
│  • 上级主管: 由发起人的上级审批                                 │
│  • 发起人自选: 发起时选择审批人                                 │
│  • 表单字段: 从表单中获取审批人                                 │
│  • 上一步处理人的上级: 动态获取                                 │
│                                                                 │
│  【审批操作】                                                    │
│                                                                 │
│  • 同意: 通过当前节点,流转到下一节点                            │
│  • 拒绝: 驳回申请,流程结束                                      │
│  • 驳回: 驳回到指定节点重新处理                                 │
│  • 转办: 转给其他人处理                                         │
│  • 加签: 增加审批人                                             │
│  • 减签: 减少审批人                                             │
│  • 撤回: 发起人撤回申请                                         │
│  • 终止: 终止流程                                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 流程定义表 (wf_process_definition)

```sql
CREATE TABLE wf_process_definition (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id           UUID NOT NULL,
    code                VARCHAR(100) NOT NULL,
    name                VARCHAR(200) NOT NULL,
    category_id         UUID,
    description         TEXT,
    icon                VARCHAR(100),
    form_id             UUID,
    form_data           JSONB,
    bpmn_xml            TEXT NOT NULL,
    variables           JSONB,
    initiator_range     JSONB,
    business_config     JSONB,
    version             INTEGER NOT NULL DEFAULT 1,
    status              VARCHAR(20) NOT NULL DEFAULT 'draft',
    published_at        TIMESTAMP,
    published_by        UUID,
    created_at          TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMP NOT NULL DEFAULT NOW(),
    created_by          UUID,
    updated_by          UUID,
    deleted_at          TIMESTAMP,
    
    UNIQUE(tenant_id, code, version)
);

CREATE INDEX idx_process_def_tenant ON wf_process_definition(tenant_id);
CREATE INDEX idx_process_def_status ON wf_process_definition(tenant_id, status);
```

#### 流程实例表 (wf_process_instance)

```sql
CREATE TABLE wf_process_instance (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id           UUID NOT NULL,
    definition_id       UUID NOT NULL,
    definition_version  INTEGER NOT NULL,
    process_code        VARCHAR(100) NOT NULL,
    process_name        VARCHAR(200) NOT NULL,
    business_key        VARCHAR(200),
    title               VARCHAR(500),
    initiator_id        UUID NOT NULL,
    initiator_name      VARCHAR(100),
    organization_id     UUID,
    variables           JSONB,
    status              VARCHAR(20) NOT NULL DEFAULT 'running',
    start_time          TIMESTAMP NOT NULL,
    end_time            TIMESTAMP,
    duration            BIGINT,
    delete_reason       VARCHAR(500),
    callback_url        VARCHAR(500),
    created_at          TIMESTAMP NOT NULL DEFAULT NOW(),
    
    FOREIGN KEY (definition_id) REFERENCES wf_process_definition(id)
);

CREATE INDEX idx_process_inst_tenant ON wf_process_instance(tenant_id);
```

```sql
CREATE TABLE wf_task_instance (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id           UUID NOT NULL,

    process_definition_id UUID,
    task_def_key        VARCHAR(100) NOT NULL,
    task_name           VARCHAR(200) NOT NULL,
    task_type           VARCHAR(20) NOT NULL DEFAULT 'user',
    description         TEXT,
    assignee_id         UUID,
```
