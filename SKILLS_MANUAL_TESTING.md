## 📋 测试用例设计方法

### 1. 等价类划分

**定义**: 将所有可能的输入数据划分为若干等价类，从每个等价类中选择少数代表性数据作为测试用例。

**示例 - 客户类型验证**:
```
有效等价类:
- ENTERPRISE (企业)
- INDIVIDUAL (个人)
- GOVERNMENT (政府)

无效等价类:
- 空值
- 未知类型
- 特殊字符
- 超长字符串

测试用例:
1. 输入 ENTERPRISE → 预期：验证通过 ✅
2. 输入 INDIVIDUAL → 预期：验证通过 ✅
3. 输入 空值 → 预期：验证失败 ❌
4. 输入 UNKNOWN → 预期：验证失败 ❌
5. 输入 @#$% → 预期：验证失败 ❌
```

### 2. 边界值分析

**定义**: 对输入输出的边界值进行测试，因为错误往往发生在边界附近。

**示例 - 订单数量验证**:
```
有效范围：1-9999

边界值:
- 最小值：1
- 略大于最小值：2
- 正常值：500
- 略小于最大值：9998
- 最大值：9999
- 略小于最小值：0
- 略大于最大值：10000

测试用例:
1. 数量=0 → 预期：失败 ❌
2. 数量=1 → 预期：通过 ✅
3. 数量=2 → 预期：通过 ✅
4. 数量=500 → 预期：通过 ✅
5. 数量=9998 → 预期：通过 ✅
6. 数量=9999 → 预期：通过 ✅
7. 数量=10000 → 预期：失败 ❌
```

### 3. 因果图法

**定义**: 通过分析输入条件（因）和输出结果（果）之间的关系，设计测试用例。

**示例 - 工单创建**:
```
输入条件:
- C1: 客户名称不为空
- C2: 服务类型已选择
- C3: 问题描述不为空

输出结果:
- E1: 创建成功
- E2: 创建失败

关系:
- C1 AND C2 AND C3 → E1
- NOT C1 OR NOT C2 OR NOT C3 → E2

测试用例:
1. C1=T, C2=T, C3=T → E1 ✅
2. C1=F, C2=T, C3=T → E2 ✅
3. C1=T, C2=F, C3=T → E2 ✅
4. C1=T, C2=T, C3=F → E2 ✅
5. C1=F, C2=F, C3=F → E2 ✅
```

### 4. 场景法

**定义**: 通过模拟用户实际使用场景来设计测试用例。

**示例 - 售后工单完整流程**:
```
基本流 (Happy Path):
1. 用户登录系统
2. 进入售后管理
3. 点击新建工单
4. 填写工单信息
5. 提交工单
6. 系统验证通过
7. 工单创建成功

备选流:
A1: 客户信息不存在
   - 系统提示"客户不存在"
   - 用户创建客户或选择其他客户
   
A2: 必填字段未填写
   - 系统提示"请填写必填项"
   - 用户补充信息后重新提交
   
A3: 网络异常
   - 系统提示"网络异常"
   - 用户检查网络后重新提交

异常流:
E1: 系统故障
   - 系统记录错误日志
   - 提示用户稍后重试
```

### 5. 错误推测法

**定义**: 基于测试人员的经验和直觉，推测系统中可能存在的错误，设计测试用例。

**示例 - 常见错误推测**:
```
1. 特殊字符输入
   - SQL 注入：' OR '1'='1
   - XSS 攻击：<script>alert('xss')</script>
   
2. 极端数据
   - 超大文件上传
   - 超长文本输入
   - 超大数量订单
   
3. 并发操作
   - 多人同时修改同一记录
   - 重复提交表单
   
4. 网络异常
   - 请求超时
   - 网络中断
   - 服务器宕机
```
